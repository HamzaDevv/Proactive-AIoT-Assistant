GUIDING PRINCIPLE: SERVERLESS-FIRST
───────────────────────────────────
Every component runs as an embedded library or in-process sub-process.
No Docker. No separate database ports. No server management.
The entire stack is `pip install` + a single `python run.py`.

  ┌─────────────────────────────────────────────────────────────────┐
  │                     Serverless Stack Summary                    │
  ├────────────────────┬────────────────────────────────────────────┤
  │  ChromaDB          │  REMOVED — replaced by Mem0 + PageIndex   │
  │  Neo4j             │  REMOVED — replaced by Kuzu (embedded)    │
  │                    │    or FalkorDBLite (sub-process, zero-cfg) │
  │  Mem0              │  Evolving user habits & preference memory  │
  │  PageIndex         │  Structured doc retrieval (manuals, rules) │
  │  SQLite            │  Device state, event logs, preferences DB  │
  │  Ollama            │  Local LLM & embeddings (already in use)   │
  │  MQTT (Mosquitto)  │  Lightweight broker for ESP32 sensors      │
  └────────────────────┴────────────────────────────────────────────┘

ARCHITECTURE OVERVIEW
─────────────────────
Three-layer cognitive model — SENSE → THINK → ACTION — with a
device-agnostic "Cognitive Linker" and a Hybrid Reasoning Engine.

Data flow:
  Sensors/APIs  ──►  ContextPacket  ──►  THINK Pipeline  ──►  SuggestionSchema
       ▲                                     │  ▲                    │
       │            Hybrid Reasoning Engine   │  │                    ▼
       │           ┌─────────────────────┐    │  │                 ACTION
       │           │  Mem0 (habits)      │◄───┘  │                    │
       │           │  PageIndex (docs)   │       │                    │
       │           │  Kuzu (causality)   │       │                    │
       │           │  SQLite (state)     │       │                    │
       └───────────│─────────────────────│◄──────└──── Feedback ◄────┘
                   └─────────────────────┘

2.1  Two-Pass LLM Pipeline                                       [DONE]

     ✓ Gemini 2.0 Flash as primary LLM

2.2  Deterministic Decision Graph                                [DONE]
     ✓ Rule-based candidate generation from context
     ✓ Temperature, time-of-day, activity-based rules

2.3  Safety Checker                                              [DONE]
     ✓ Device blacklist, capability validation, parameter range checks

2.4  Proactivity Budget                                          [DONE]
     ✓ Cooldown-based throttle to prevent suggestion fatigue

─── NEW: HYBRID REASONING ENGINE (replaces ChromaDB) ──────────────

2.5  Mem0 — Evolving User Habit Memory                           [TODO]
     WHY: Mem0 is a self-improving memory layer that automatically
     extracts, stores, and evolves user preferences from conversations
     and interactions. Unlike flat vector search, it understands that
     preferences CHANGE over time and updates itself.

     SETUP (fully local, no cloud):
     • pip install mem0ai
     • Configure with Ollama as BOTH the LLM and embedding provider
       (no OpenAI API key needed):
         config = {
             "llm": {"provider": "ollama", "config": {"model": "llama3"}},
             "embedder": {"provider": "ollama",
                          "config": {"model": "nomic-embed-text"}},
             "vector_store": {"provider": "qdrant",
                              "config": {"collection_name": "sadaf_memory",
                                         "path": "./mem0_qdrant"}}
         }
         m = Memory.from_config(MemoryConfig(**config))
     • Alternatively use "lancedb" as vector store (pure Python,
       zero-config, stores as Parquet files on disk)

     ARCHITECTURE:
     ├── User Memory: per-user preference profiles that evolve
     │   "Hamza likes 22°C while working, 24°C while sleeping"
     │   "Mom prefers bright warm lights in the kitchen"
     ├── Session Memory: current conversation context and intent
     ├── Agent Memory: learned patterns across all interactions
     │   "When outdoor temp > 35°C, users always want AC at 22°C"
     └── Automatic Conflict Resolution:
         Mem0 handles contradictions — if user says "I like it cool"
         but later says "24°C is fine", it updates, not duplicates

     INTEGRATION POINTS:
     • ThinkOrchestrator.process_context() calls mem0.search()
       instead of ChromaStore.get_relevant_info()
     • ThinkOrchestrator.store_user_confirmation() calls mem0.add()
       instead of ChromaStore.add_document()
     • mem0.update() called when user overrides AI decisions

2.6  PageIndex — Structured Document Retrieval                   [TODO]
     WHY: For device manuals, safety rulebooks, and appliance specs,
     vector search gives ~50% accuracy ("vibes-based retrieval").
     PageIndex uses hierarchical document trees and LLM reasoning
     to achieve 98.7% accuracy on complex structured documents.

     SETUP:
     • pip install pageindex
     • Feed structured PDFs: device manuals, safety protocols,
       appliance technical specs, home wiring diagrams

     HOW IT WORKS:
     ┌─────────────────────────────────────────────────────┐
     │ Traditional RAG:  chunk → embed → similarity search │
     │ Result: "The AC manual says..." (maybe wrong page)  │
     │                                                     │
     │ PageIndex:  PDF → hierarchical tree → LLM navigates │
     │ Result: exact page, exact spec, traceable path      │
     └─────────────────────────────────────────────────────┘

     WHAT GOES INTO PAGEINDEX:
     ├── Device technical manuals (from manufacturers)
     │   → "What is the max safe temp for smart_geyser_1?" → exact °C
     ├── Safety rulebooks & building codes
     │   → "Can induction run unattended?" → deterministic answer
     ├── EUPont ontology documentation
     │   → device capability mapping reference
     └── Appliance wiring / installation guides

     INTEGRATION:
     • SafetyChecker queries PageIndex BEFORE allowing actions
     • IoTGPT Derive stage uses PageIndex to look up exact device
       capabilities instead of relying on LLM hallucination
     • Answers are traceable: "Based on manual page 47, section 3.2"

2.7  Embedded Graph DB — Causal Reasoning & Event Clock          [TODO]

     OPTION A: Kuzu (RECOMMENDED for analytics)
     ─────────────────────────────────────────
     • pip install kuzu
     • Embedded C++ engine, runs in-process (like SQLite for graphs)
     • Stores as a folder on disk — Git-friendly, copyable, movable
     • Cypher query language, 18x faster data ingestion than Neo4j
     • Native LangChain integration (KuzuGraphQA)
     • Best for: complex multi-hop analytical queries on event chains

     OPTION B: FalkorDBLite (RECOMMENDED for simplicity)
     ─────────────────────────────────────────────────────
     • pip install falkordblite
     • Launches a lightweight sub-process via Unix domain sockets
     • Zero config, zero open ports, process isolation (crash-safe)
     • Full Cypher support, identical API to full FalkorDB
     • Migration path: swap one import to move to FalkorDB Cloud
     • Best for: prototyping, simpler graph needs, easy setup

     THE EVENT CLOCK (what the graph stores):
     ├── Nodes: User, Device, Action, Sensor, Policy, Environment
     ├── Edges: TRIGGERED_BY, APPLIED_POLICY, CAUSED, OVERRIDDEN_BY
     └── Enables:
         • "Why did the AC turn on at 3 AM?" → graph traversal shows
           the full causal chain: sensor→policy→action
         • "Show all times Hamza overrode a lighting decision" →
           relationship query feeds back into Mem0 to update prefs
         • Multi-occupant conflict detection via relationship analysis

     HYBRID SEARCH PATTERN:
     ┌──────────────────────────────────────────────────────────┐
     │ User query: "Make the room comfortable"                  │
     │                                                          │
     │ 1. Mem0.search() → "Hamza likes 22°C, 60% humidity"     │
     │ 2. PageIndex → "AC max cooling: 16°C, geyser max: 75°C" │
     │ 3. Graph → "Last 3 times at this time, AC was set to 22" │
     │ 4. LLM fuses all 3 → confident, grounded suggestion     │
     └──────────────────────────────────────────────────────────┘

PHASE 5 — SECURITY & PRIVACY (Local-First, Zero Cloud)
───────────────────────────────────────────────────────
Goal: All data stays on-device. Secrets are injected just-in-time
      and immediately scrubbed.

5.1  Local-First Processing                                      [TODO]
     • Primary inference: Apple Silicon (unified memory) or similar
     • SLMs via Ollama (Llama-3 / Mistral) for 90% of tasks
     • Cloud escalation (Gemini) only for complex reasoning
     • All local files encrypted at rest: AES-256
       - SQLite: SQLCipher extension for transparent encryption
       - Mem0 vector store: encrypted disk volume or file-level
       - Kuzu/FalkorDB data folder: encrypted at filesystem level

5.2  Vault Layer — Just-in-Time Secret Injection                 [TODO]
     • Lightweight vault (HashiCorp Vault or simpler self-hosted):
       - Stores: API keys, user profiles, OAuth tokens
       - Just-in-Time pattern:
         1. Agent requests secret → vault authenticates agent (DID)
         2. Vault injects credential into the specific tool call
         3. Tool executes → vault immediately scrubs secret from
            the LLM's context window and memory
         4. Comprehensive audit log of every access
     • For simpler setups: python-dotenv with encrypted .env files
       using age or sops for git-safe secret management

5.3  Zero-Trust Agent Authentication                             [TODO]
     • DIDs for each AI agent — cryptographic identity proof
     • OIDC flows for secure token exchange (least privilege)
     • MCP tool scoping: each agent sees only its required tools

5.4  Safety Guardrails (Non-LLM, Deterministic)                  [PARTIAL]
     ✓ SafetyChecker with blacklist + parameter validation
     • PageIndex-backed safety rules:
       SafetyChecker queries PageIndex with "Is it safe to [action]?"
       → gets deterministic answer from the actual device manual
       → NOT from the LLM's training data
     • Hard-coded rules that can NEVER be overridden by the LLM:
       - No heating appliances when occupancy = 0
       - Human approval required for door unlock
       - Max geyser temp = 60°C regardless of user request
       - Smoke/CO detection → immediate alert, suppress nothing
